# Autospy

![Autospy icon](autospy_icon_192.png)

App icon is available in SVG and PNG at 32, 64, 96, and 192px.

Scheduled Python tool that scrapes Carvana for used SUVs, scores and filters listings with AI analysis, and sends an HTML email summary with price trend charts.

Comes with a full **admin dashboard** — a desktop app or local web UI — for managing search profiles, triggering runs, previewing emails, and viewing history.

---

## Quickstart — already have the installer?

If someone has shared the `Autospy_0.1.0_x64-setup.exe` file with you, this is all you need:

1. **Run the installer** — double-click `Autospy_0.1.0_x64-setup.exe` and follow the prompts.
2. **Launch Autospy** from the Start menu or desktop shortcut. A small icon appears in your system tray (bottom-right corner of the taskbar).
3. **Open the dashboard** — double-click the tray icon, or right-click it and choose **Open Dashboard**.
4. **Set up Gmail** — go to the **Setup** page. If the Gmail row is red, click **Run OAuth setup**, sign in with your Google account, and follow the prompts. This only needs to be done once.
5. **Create a search profile** — go to the **Profiles** page and click **+ New profile**. Fill in the vehicles you want to track, your price/mileage limits, and your email address.
6. **Run a search** — go to the **Run** page, select your profile, and click **Run now**. Results will be emailed to you when it finishes.

> **Note:** Autospy requires Python to be installed on the machine. If the dashboard doesn't load after launching, make sure [Python 3.11+](https://www.python.org/downloads/) is installed and the project dependencies have been set up (`pip install -r requirements.txt` from the project folder).

## Dashboard — getting started

The dashboard is the main way to use this tool. There are two ways to open it depending on how it was installed.

### Option A — Desktop app (Tauri, recommended)

A Windows installer is generated by the build process. To build it yourself:

```bash
cd dashboard/frontend
npm install

# Debug build (faster, includes logging)
npx @tauri-apps/cli build --debug

# Release build (optimised, for distribution)
npx @tauri-apps/cli build
```

Installers are written to:
```
# Debug
dashboard/frontend/src-tauri/target/debug/bundle/nsis/Autospy_0.1.0_x64-setup.exe
dashboard/frontend/src-tauri/target/debug/bundle/msi/Autospy_0.1.0_x64_en-US.msi

# Release
dashboard/frontend/src-tauri/target/release/bundle/nsis/Autospy_0.1.0_x64-setup.exe
dashboard/frontend/src-tauri/target/release/bundle/msi/Autospy_0.1.0_x64_en-US.msi
```

Use the NSIS `.exe` for most installations. Use the release build for distribution; use the debug build if you need verbose logs to diagnose issues.

Run the installer, then launch **Autospy** from the Start menu or desktop shortcut. The app places an icon in your system tray (bottom-right of the taskbar). From there:

- **Double-click** the tray icon (or left-click) to open the dashboard window
- **Right-click → Open Dashboard** to bring it back if you minimised it
- **Right-click → Quit** to fully close the app (closing the window just hides it to the tray)

The dashboard window opens automatically when the app launches.

### Option B — Browser-based (no installer needed)

If you have Python and the project checked out, you can run the dashboard directly in any browser.

**Step 1 — Start the backend:**

```bash
cd car_search
uvicorn dashboard.backend.app:app --host 127.0.0.1 --port 8000
```

**Step 2 — Open the dashboard:**

Visit **http://localhost:8000** in your browser. The React frontend is served as static files from the same backend.

**For development** (with hot reload), run the Vite dev server instead:

```bash
# Terminal 1
uvicorn dashboard.backend.app:app --reload --host 127.0.0.1 --port 8000

# Terminal 2
cd dashboard/frontend
npm run dev        # opens at http://localhost:5173
```

## Dashboard views

### Run

The main page. Select which profiles to run (leave all unchecked to run everything), pick options, and click a button.

| Button | What it does |
|---|---|
| **Run now** | Full scrape → LLM analysis → save results → send email |
| **Dry run + preview** | Full scrape and analysis, no email sent, renders an email preview inline |
| **Cancel** | Stops the running job |

The log terminal streams live output as the job runs. When a dry run finishes, the rendered email HTML appears below the log.

**Options:**

| Option | Description |
|---|---|
| LLM backend — Auto | Try Ollama first, fall back to Anthropic API |
| LLM backend — Ollama | Force Ollama only |
| LLM backend — API | Force Anthropic API only |
| LLM backend — No LLM | Skip AI analysis entirely |
| Force email | Send email even if no new listings or price drops |
| Suppress email | Don't send email this run |
| Debug logging | Show verbose DEBUG output in the log terminal |

### Profiles

Create, edit, and delete search profiles. Each profile defines:

- Which vehicles to search (one or more make/model pairs)
- Price, mileage, and year filters
- Fuel type (gas, hybrid, or all)
- Email recipients
- Trim exclusions, model preference order
- Whether to show estimated financing
- Which reference doc to feed to the LLM

Click **+ New profile** or **Edit** on an existing one to open the profile form.

### History

Shows all past runs and pricing trends.

- **Metric cards** — total runs, unique VINs seen, models tracked, cheapest ever listing
- **Runs table** — date, listings saved vs. found, LLM backend used, run duration
- **Price trends** — line charts of average and minimum price per model over the last 30/60/90 days

### Docs

Manage the vehicle reference markdown files in `vehicle_reference/`. These files are fed to the LLM to improve analysis quality for specific models.

Name files `make_model.md` (e.g. `honda_crv.md`) for auto-discovery — the tracker will automatically match them to the right vehicle without any configuration. The file list shows which profiles each doc is currently matched to.

### Setup

Health check for all system components with action buttons for anything that needs attention.

| Component | Green means... |
|---|---|
| Profiles | `profiles.yaml` exists with at least one profile |
| Ollama | Ollama is reachable and a model is available |
| Anthropic API | `ANTHROPIC_API_KEY` is set in `.env` |
| Gmail | OAuth token is present and valid |
| Playwright | Chromium browser is installed |

If Playwright is missing, an **Install Chromium** button appears. If Gmail isn't configured, a **Run OAuth setup** button walks you through it — both stream their output live in the page.

### Settings

Edit all non-secret settings in one place. Changes are saved to `dashboard_settings.json` and take effect on the next run.

Secret values (API keys, OAuth credentials) are shown as `***` and can be updated by typing a new value — leave them blank to keep the existing value.

## One-time setup

### Prerequisites

- Python 3.11+
- Node.js 18+ and npm (for the dashboard frontend)
- For the Tauri desktop app: [Rust](https://rustup.rs/) (install via `rustup`)

### 1. Clone and install Python dependencies

```bash
git clone <repo-url>
cd car_search
pip install -r requirements.txt
playwright install chromium
```

Or install Chromium from the **Setup** page in the dashboard after the backend is running.

### 2. Create your `.env` file

```bash
cp .env.example .env
```

Fill in your values:

```dotenv
# Required for AI analysis
ANTHROPIC_API_KEY=sk-ant-...

# Required for email (see Gmail OAuth setup below)
GMAIL_SENDER=you@gmail.com
GMAIL_CLIENT_ID=
GMAIL_CLIENT_SECRET=
GMAIL_REFRESH_TOKEN=          # written automatically by setup or the Dashboard Setup page

EMAIL_FROM_NAME=Autospy

# Optional — network Ollama server(s)
# OLLAMA_NETWORK_HOST=192.168.0.100:11434
# OLLAMA_NETWORK_HOST_2=192.168.0.101:11434
```

### 3. Set up Gmail OAuth (one time)

The tracker sends email through the Gmail API. You need a Google Cloud project.

**Create OAuth credentials:**

1. Go to [console.cloud.google.com](https://console.cloud.google.com) → select or create a project
2. **APIs & Services → Enable APIs → Gmail API → Enable**
3. **APIs & Services → Credentials → Create Credentials → OAuth client ID → Desktop app**
4. Copy the **Client ID** and **Client Secret** into your `.env`

**Authorise the app** — pick one:

- **From the dashboard:** open the Setup page → click **Run OAuth setup**
- **From the terminal:** `python setup_gmail_oauth.py`

Both open a browser, ask you to sign in, and write `GMAIL_REFRESH_TOKEN` to your `.env` automatically.

### 4. Create your first search profile

Open the dashboard → **Profiles** → **+ New profile**. Fill in the vehicles you want to track, your filters, and at least one email recipient.

Alternatively, edit `profiles.yaml` directly — see the [profiles.yaml reference](#profilesyaml-reference) below.

### 5. Verify everything

Open the dashboard → **Setup** — all indicators should be green (or "not configured" for services you're not using).

Or from the terminal:

```bash
python main.py --check-setup
```

## Developer / CLI usage

The tracker can be run entirely from the command line without the dashboard.

```bash
# Full run — scrape, analyse, save, email
python main.py

# Dry run — no save, no email
python main.py --dry-run

# Skip LLM analysis (rule-based scoring only)
python main.py --no-llm

# Force a specific LLM backend
python main.py --backend api
python main.py --backend ollama

# Force email send regardless of change detection
python main.py --email

# Suppress email for this run
python main.py --no-email

# Run specific profiles only
python main.py --profile my_search suv_search

# Run on a schedule (every check_interval_hours)
python main.py --schedule

# Print run history, per-model stats, and pricing trends
python main.py --history

# Validate config and test all backends without scraping
python main.py --check-setup

# Show DEBUG-level output
python main.py --debug
```

### Running the backend only (without the Tauri app)

```bash
uvicorn dashboard.backend.app:app --reload --host 127.0.0.1 --port 8000
```

The `--reload` flag restarts the server when Python files change. Remove it for a stable production instance.

### Running the tray helper (Phase 1 launcher)

If you want the tray icon without building the full Tauri app:

```bash
python dashboard/tray/tray_app.py
```

This starts the backend and adds a tray icon. Set `DASHBOARD_URL=http://localhost:5173` before running if you want it to point at the Vite dev server instead.

## How it works

Each run:

1. Loads one or more **search profiles** from `profiles.yaml`
2. Scrapes Carvana using headless Chromium across configured fuel types
3. Deduplicates by VIN and applies per-profile filters (price, mileage, year, trim exclusions)
4. Scores each listing 0–100 (see [Value score](#value-score))
5. Runs **per-make LLM analysis** — one call per make, each with its own reference doc, to prevent cross-brand terminology bleed
6. Runs a **cross-model synthesis** LLM call to rank and recommend the top 3 picks across all makes
7. Validates LLM output for brand-bleed errors; auto-corrects if possible
8. Saves results to a timestamped CSV and SQLite history database
9. Detects new listings and price drops since the last run
10. Sends an HTML email with the top listings table, price trend charts, trim key, and AI analysis

## profiles.yaml reference

Multiple profiles run in a single invocation, each with its own vehicles, filters, and recipients.

```yaml
profiles:
  - profile_id: my_search          # unique slug — letters, numbers, underscores only
    label: "My SUV Search"         # shown in email subject and body

    vehicles:                      # [make, model] pairs to search
      - [Honda, CR-V]
      - [Toyota, RAV4]

    max_price: 30000               # omit or null for no upper limit
    max_mileage: 80000
    min_year: 2021
    max_year: 2025
    excluded_years: [2022]         # specific years to skip within the min/max range

    fuel_type_filters: [Hybrid, Gas]   # Hybrid | Gas | null (all).
                                       # Runs a separate Carvana search per type.

    model_preference: [CR-V, RAV4]     # ordered best→worst; affects sort and value score

    excluded_trim_keywords: [sport]    # case-insensitive substrings to drop

    show_financing: true               # show estimated monthly payments in email
    down_payment: 3000                 # overrides the global down_payment setting

    reference_doc_path: null           # null = auto-discover from vehicle_reference/
                                       # or set to ./vehicle_reference/my_doc.md for a
                                       # single combined doc for the whole profile

    email_to:
      - you@gmail.com
```

## Value score

Each listing is scored 0–100 before LLM analysis. Higher is better.

| Component | Weight | Logic |
|---|---|---|
| Price vs. group average | 35 | % below average price for same make/model, capped at ±30% |
| Mileage | 25 | Inverse linear: 0 mi = 25 pts, `max_mileage` = 0 pts |
| Age | 20 | Newer = better. `max_year` = 20 pts, `min_year` = 0 pts |
| Hybrid bonus | 10 | +10 if `fuel_type_filters` includes Hybrid and trim qualifies |
| Shipping penalty | 10 | 10 pts if no shipping fee; scales to 0 at $1,500 |

`model_preference` adds a tie-breaking sort on top of the raw score. LLM top picks are pinned to the top rows of the email table.

## AI analysis

### Architecture

Analysis runs in three phases:

1. **Per-make analysis** — One LLM call per distinct make. Each call receives only that make's listings and its own reference doc, preventing brand terminology from bleeding across makes.

2. **Cross-model synthesis** — One final LLM call sees the combined listing table and the per-make summaries. It produces the top 3 picks and a single final recommendation. This is what appears in the email body.

3. **Validation** — Output is checked for brand-bleed errors (e.g. "EyeSight" in a non-Subaru context). A correction pass runs automatically if issues are found. A warning banner is injected into the email if issues remain after correction.

### Backend selection

Backends are tried in this order:

1. **Ollama (network)** — Free, private, runs on your own server. Set `ollama_enabled: true` in Settings and `OLLAMA_NETWORK_HOST` in `.env`. If two hosts are configured the tracker routes to whichever responds faster.
2. **Anthropic API** — Higher quality. Costs fractions of a cent per run. Used automatically if Ollama is unavailable or disabled.
3. **None** — If both fail the run completes without AI analysis. Scoring and email still work.

### Reference docs

Markdown files in `vehicle_reference/` are automatically matched to each make/model by filename. A file named `honda_crv.md` or `crv_reference.md` both match a Honda CR-V search.

Each file can include trim hierarchy, known issues at specific mileage/year ranges, hybrid vs. gas trade-offs, and what a good deal looks like at current market prices.

## Known limitations

- **Bot detection:** Carvana uses PerimeterX. Headless Chromium passes most of the time. If scraping fails consistently, residential proxies are the next step.
- **Shipping costs:** Often unavailable in search results. Shown as `None` in the CSV when missing.
- **Synthesis cap:** The cross-model synthesis call is capped at 30 listings. Listings beyond position 30 by score won't be eligible for top-pick selection. Per-make analyses still see all listings.
- **No official Carvana API:** The scraper can break if Carvana changes their frontend. The three-strategy extractor (Next.js → Apollo → DOM fallback) is designed to be resilient but may need maintenance.
- **Prices change constantly.** Each CSV row is a snapshot at scrape time.
- **Tauri sidecar requires Python in PATH.** The desktop app spawns `python -m uvicorn` directly. Python and all pip dependencies must be installed on the machine running the app. A fully self-contained bundle (PyInstaller) is a future improvement.
